# AGENTS.md

This file provides guidance to WARP (warp.dev) when working with code in this repository.

## Essential commands
- Environment: duplicate `.env.example` to `.env` (or `.env.dev`) before running CLI commands; most targets assume `MONGO_URL`, `MONGO_DATABASE`, and `MIGRATIONS_COLLECTION` are defined. `make dev-setup` installs `golangci-lint` and `goimports`, while `make deps` runs `go mod download` + `tidy` across the workspace (Go 1.25 compatibility per `makefiles/variables/vars.mk`).
- Build & smoke-test: `make build` emits `build/mongo` (binary aliases `mt`/`mmt`). Run `./build/mongo version` or `./build/mongo status --offline` to confirm wiring without a database. `make build-all` cross-compiles for darwin/linux/windows.
- Lint/format: `make lint` wraps `golangci-lint run ./...`; `make format` and `make vet` live in `makefiles/linter.mk`. `make pr-check` sequences build → lint → unit tests → integration tests.
- Unit tests: `make test` runs all non-example packages, and `make test-coverage` also writes `coverage.html`. For a focused loop, use plain `go test ./internal/migration -run TestPlan_WithTarget` (any package path works). Table-driven tests are the expected style (documented in `README.ai.md`).
- Integration tests: `make integration-test` builds multi-arch binaries, spins up `docker/integration-compose.yml`, and runs `go test -tags=integration ./integration-tests` against a MongoDB container on `INTEGRATION_MONGO_PORT` (default 37017); the target tears the stack down afterward. Use `make db-up` / `make db-down` for ad-hoc local databases.
- Migration workflow: after `make build`, use `make create-migration DESC="add email index"` to scaffold a timestamped file under `migrations/`, `make migration-status` to inspect applied versions, `make migration-up` to run pending migrations, and `make migration-down VERSION=20240101_001` for targeted rollback. The CLI mirrors these (`./build/mongo up`, `down --target`, `status`, `create`).
- MCP workflows: `make mcp` starts the stdin/stdout MCP server with your current config; `make mcp-examples` registers the sample migrations from `migrations/`. `make mcp-test` boots Mongo in Docker and runs the MCP integration suite; `make mcp-client-test` lets you issue raw JSON-RPC requests. Generate assistant config via `./build/mongo mcp config` or `make mcp-config` (copies the same JSON).
- Docker & scripts: `make docker-build`/`docker-run` rely on `docker/compose.yml`. Ops scripts live in `scripts/` (e.g., `deploy-migrations.sh auto` via `make deploy-dev` / `make deploy-prod`). `start.sh`/`start` are convenience wrappers for supervisord-based demos.
- Release: `make release-check` performs a dry run (`goreleaser release --snapshot`), while `make release` / `make release-beta` invoke the real pipeline. GitHub Actions `.github/workflows/release.yml` logs in to GHCR, builds all artifacts, and also updates the Homebrew tap; publish jobs require `HOMEBREW_TAP_TOKEN` (a PAT with `repo` scope on `drewjocham/homebrew-mongork`). The recent failure (`401 Bad credentials`) means the secret is missing or lacks access—double-check before tagging to avoid reruns. Post-release, `post-release` job pokes `proxy.golang.org` so pkg.go.dev refreshes immediately.

## Architecture overview
- CLI entrypoint: `cmd/main.go` simply calls `internal/cli.Execute()`. `internal/cli/root.go` wires Cobra commands (`up`, `down`, `status`, `create`, `schema`, `oplog`, MCP, etc.) and injects a `Services` struct (config, `migration.Engine`, and `mongo.Client`) into the command context so subcommands pull shared dependencies via typed keys in `context.Context`. Logging is centralized through `internal/log` (Zap) and respects `--debug` / `--log-file`. Commands marked with the `annotationOffline` flag skip dialing Mongo (e.g., `mcp config`, `status --offline`).
- Config & bootstrap: `internal/config` consolidates `.env` files and environment variables, exposes helpers like `GetConnectionString()`/`Validate()`, and stores TLS + pool tuning. `bootstrap()` (in `root.go`) loads config, optionally prints it (`--show-config`), then calls `dial()` which wraps the MongoDB Go Driver v2 (`go.mongodb.org/mongo-driver/v2`) with ping retries and TLS options. Every component downstream pulls configuration from the context, so new commands should avoid re-reading the filesystem.
- Migration engine & registries: `internal/migration/engine.go` is the authoritative pipeline—distributed locking via the `migrations_lock` collection, per-migration SHA256 checksums (`validateChecksum`), and planning logic (`Plan`, `run`, `executeWithRetry`) that supports transactional retries when the server allows it. Migrations self-register inside `migrations/register.go` through `migration.MustRegister`, which enforces the timestamped naming convention defined in `internal/migration/registry.go`. Schematized index metadata lives in `internal/schema` and is populated by `migrations/indexes.go`; CLI commands and MCP’s `database_schema` tool read that registry to render index tables.
- Parser, JSON, and payload tooling: The CLI’s `parse`/`validate` commands and the MCP `parse_payload`/`validate_payload` tools compose `internal/parser`, `internal/model`, and `internal/jsonutil`. `parser.DecodePayload` autodetects JSON/BSON, `ParseInto` applies optional validation against registry entries, and `model.ParseByType` routes documents based on discriminator fields so you can add new payload types by registering constructors. `jsonutil` hides the standard library marshal/unmarshal to guarantee consistent map ordering and indentation—stick with it whenever emitting JSON back to MCP clients.
- MCP server stack: `internal/cli/mcp.go` launches `mcp.NewMCPServer` (see `mcp/server.go`) with the current config, logs over stderr to keep stdin/stdout clean for the protocol, and optionally calls `registerExampleMigrations()` so AI clients can experiment without touching production data. `mcp/handlers.go` registers the toolset (status/up/down/create/schema/parse/validate) with `github.com/modelcontextprotocol/go-sdk/mcp`, lazily connects to Mongo on demand, and reuses the same migration engine used by the CLI. Handlers rely on parser + schema registries described above, so adding a new tool typically means stitching together those packages rather than duplicating logic.
- Testing & integration layers: Unit suites cover the migration engine (`internal/migration/engine_test.go`), parser/model cleanup, and CLI helpers. Integration tests live under `integration-tests/` and run with the `integration` build tag; `make integration-test` ensures Docker Compose builds the CLI image (`docker/integration-compose.yml`) before kicking off the Go test package. The MCP integration tests (`mcp` package with the same tag) rely on Mongo containers spun up during `make mcp-test`. Follow the table-driven style from `README.ai.md`, keep mocks minimal, and prefer the shared helpers already in the repo over bespoke harnesses.
- Documentation & examples: `readme.md` is the user-facing quick start; `install.md`, `library.md`, and `mcp.md` dive into platform installs, embedding the engine in other Go programs, and wiring MCP clients (Ollama, Claude Desktop, Goose, etc.). `doc.go` feeds pkg.go.dev, and the `examples/` directory (including `examples/library-example`) shows canonical registration + execution patterns. Keep these docs in sync whenever you introduce new commands or configuration flags so future agents can rely on them instead of re-deriving behavior from source.
